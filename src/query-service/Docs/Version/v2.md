# Query Service v2 - Guarded Tool Routing & MCP Client

## Mục tiêu

Phiên bản v2 đưa `query-service` về đúng hướng kiến trúc hệ thống: Query Service là **LLM orchestration + MCP client**, để LLM chọn tool `rag_search` hoặc `hr_query`, còn backend giữ các chốt an toàn bằng code.

So với bản v1 đã push trên git (`origin/dungpq/query-service`, commit `79301a5`), v1 route câu hỏi bằng `_detect_intent()` hard-code trong `QueryOrchestrationUseCase`. Cách đó nhanh và dễ test, nhưng lệch kiến trúc vì Query Service tự phân loại intent bằng keyword thay vì để agent/tool-calling chọn tool từ MCP.

V2 không dùng thêm một LLM thứ hai để validate. Validation là deterministic guard:

- Chỉ cho phép tool trong allowlist: `rag_search`, `hr_query`.
- `hr_query.intent` chỉ nhận `leave_balance`, `leave_requests`, `payroll`.
- `user_id` luôn lấy từ authenticated user/JWT, không nhận từ LLM.
- `document_ids` luôn lấy từ ACL projection, không nhận từ LLM.
- `top_k` cố định/clamp ở backend.
- Tool lạ, intent sai, decision lỗi hoặc tool unavailable đều fallback về `rag_search`.

## Luồng xử lý mới

1. `POST /query` nhận `question` và `user_id`, xác thực user như cũ.
2. `QueryOrchestrationUseCase` lưu user message và lấy recent conversation context.
3. Câu identity như “Bạn là ai?” vẫn trả lời trực tiếp bằng canned response, không gọi tool.
4. Với câu hỏi còn lại, Query Service gọi `ToolDecisionClient.choose_tool()` để LLM/mock chọn tool.
5. Backend normalize decision:
   - `hr_query` hợp lệ -> gọi MCP tool với `user_id` từ auth và `intent` đã validate.
   - Mọi decision khác -> đi `rag_search`.
6. Với `rag_search`, backend lấy `allowed_doc_ids` từ `DocumentAccessRepository`, check semantic cache, rồi gọi MCP tool với `query`, `document_ids`, `top_k=5`.
7. Kết quả RAG vẫn bị lọc bằng `RAG_SCORE_THRESHOLD`; score thấp thì fallback “Không tìm thấy thông tin trong tài liệu nội bộ.”
8. Response SSE giữ nguyên format `token` và event cuối `done/sources/session_id`.

## Thay đổi so với v1 đã push

### 1. Orchestration không còn tự route bằng keyword

V1 dùng `_detect_intent()` trong `orchestration.py` để trả về `identity`, `hr:*` hoặc `rag`.

V2 bỏ `_detect_intent()` khỏi flow chính. `orchestration.py` có `_choose_tool()`:

- gọi `mcp_client.list_tools()`;
- gọi `tool_decision_client.choose_tool()`;
- normalize decision bằng `normalize_tool_decision()`;
- fallback về `rag_search` nếu decision không hợp lệ.

### 2. Thêm contract tool decision

File mới `app/application/tool_decision.py` định nghĩa:

- `ToolDecision`;
- `VALID_TOOL_NAMES`;
- `VALID_HR_INTENTS`;
- `normalize_tool_decision()`.

Đây là lớp guard chính để không tin raw output từ LLM.

### 3. Thêm ToolDecisionClient

File mới `app/infrastructure/external/tool_decision_client.py` có:

- `MockToolDecisionClient`: dùng cho test/dev, có thể force decision trong test.
- `OpenAIToolDecisionClient`: dùng OpenAI để trả JSON chọn tool.

Trong production path, LLM chỉ chọn tool và HR intent. LLM không được sinh `user_id`, `document_ids`, `top_k` hay dữ liệu phân quyền.

### 4. Thêm MCP JSON-RPC adapter

V1 chỉ có `MockMCPClient`.

V2 thêm `MCPJsonRpcClient` trong `app/infrastructure/external/mcp_client.py`:

- `tools/list` để lấy danh sách tool;
- `tools/call` với `rag_search`;
- `tools/call` với `hr_query`;
- map MCP result về DTO `SearchResult` và `HrQueryResult`.

`MCP_MODE=mock` vẫn dùng mock. `MCP_MODE=mcp` dùng `MCPJsonRpcClient` gọi `MCP_SERVICE_URL/mcp`.

### 5. Dependency injection đổi theo config

`interfaces/api/dependencies.py` hiện chọn:

- MCP client: `MockMCPClient` hoặc `MCPJsonRpcClient`;
- tool decision client: `MockToolDecisionClient` hoặc `OpenAIToolDecisionClient`.

`reset_state_for_tests()` cũng được chỉnh để reset an toàn với cả mock client và client thật.

### 6. Intent classifier cũ chuyển thành legacy/dev

Các file `intent_classifier.py`, `intent_ai_client.py` và config `INTENT_*` hiện không còn là router production chính. Chúng được giữ như legacy/dev support và test riêng, nhưng flow chính của `QueryOrchestrationUseCase` đã chuyển sang guarded tool decision.

## Kiểm thử

Đã bổ sung/giữ test cho:

- HR decision hợp lệ gọi `hr_query` và backend inject `user_id` từ authenticated user.
- LLM/mock decision cố truyền `user_id` khác bị bỏ qua.
- RAG decision cố truyền `document_ids`/`top_k` bị bỏ qua; backend dùng ACL và `top_k=5`.
- Tool lạ fallback về `rag_search`.
- HR intent lạ fallback về `rag_search`.
- User thường không thấy tài liệu top secret.
- Semantic cache không leak source giữa các scope ACL.
- MCP JSON-RPC client map được `tools/list`, `rag_search`, `hr_query`.
- Legacy `HybridIntentClassifier` vẫn có test riêng, nhưng không đại diện cho production routing chính của v2.
