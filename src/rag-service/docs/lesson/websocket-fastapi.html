1. WebSocket là gì?

WebSocket là giao thức giao tiếp hai chiều, persistent và full-duplex giữa client và server. Nó phù hợp khi server cần đẩy dữ liệu xuống ngay lập tức hoặc khi cả hai bên phải gửi dữ liệu liên tục qua cùng một kết nối.

REST

Client gửi request, server trả response, vòng đời ngắn, phù hợp cho CRUD và API truyền thống.

WebSocket

Client mở kết nối, server accept, sau đó hai bên gửi nhận message liên tục trên cùng một channel.

Tiêu chí REST WebSocket SSE Long Polling
Chiều giao tiếp Client → Server Hai chiều Server → Client Gần real-time
Server push Không tự nhiên Rất tốt Rất tốt Giả lập
Dùng cho chat Không tối ưu Rất phù hợp Chỉ nhận tốt Tạm được
Dùng cho collaborative editing Không phù hợp Rất phù hợp Không đủ Không phù hợp
Quy tắc giảng dạy dễ nhớ: REST cho CRUD, SSE cho stream một chiều, WebSocket cho tương tác hai chiều theo thời gian thực.

2. WebSocket trong FastAPI

FastAPI hỗ trợ WebSocket trực tiếp qua decorator `@app.websocket(...)`. Ở tầng dưới, FastAPI dựa trên Starlette, ASGI và Uvicorn.

1 Client gửi request có `Upgrade: websocket`.
2 Server kiểm tra route, auth, origin và policy.
3 Server gọi `accept()` để chuyển sang trạng thái mở.
4 Hai bên gửi, nhận và xử lý message trong thời gian dài.
5 Khi mất kết nối hoặc đóng tab, server phải cleanup đúng cách.
Minimal endpoint
from fastapi import FastAPI, WebSocket

app = FastAPI()

@app.websocket("/ws")
async def websocket_endpoint(websocket: WebSocket):
    await websocket.accept()
    await websocket.send_text("Hello")
Echo loop
from fastapi import WebSocketDisconnect

@app.websocket("/ws")
async def websocket_endpoint(websocket: WebSocket):
    await websocket.accept()

    try:
        while True:
            message = await websocket.receive_text()
            await websocket.send_text(f"Echo: {message}")
    except WebSocketDisconnect:
        print("Client disconnected")

3. Các khái niệm cốt lõi cần nắm

Handshake

Connection khởi đầu bằng HTTP rồi được nâng cấp sang WebSocket bằng `101 Switching Protocols`.

Heartbeat

Dùng ping/pong để phát hiện các kết nối chết ngầm do mất mạng, sleep hoặc timeout ở proxy.

State management

Muốn broadcast, room hoặc online/offline tracking thì phải quản lý connection tập trung.

Broadcast, unicast, room

Broadcast gửi cho tất cả, unicast gửi cho một user, room gửi cho một nhóm theo channel logic.

Authentication

Auth trên WebSocket cần thiết kế riêng, có thể dùng cookie, query token, first-message auth hoặc subprotocol.

Một demo chat chỉ cần `accept()` và `receive_text()`. Một hệ production cần thêm auth, heartbeat, rate limit, validation, cleanup và sync state giữa nhiều worker.

4. Các pattern triển khai thực tế

Connection manager

Giữ danh sách socket đang mở, phục vụ broadcast và cleanup.

Room / Channel

Nhóm client theo ngữ cảnh như chat room, trading pair hoặc dashboard.

Redis Pub/Sub

Đồng bộ message giữa nhiều worker hoặc nhiều instance app.

Snapshot + delta

Khôi phục state đúng sau reconnect, tránh mất dữ liệu trên stream.

Connection manager
class ConnectionManager:
    def __init__(self):
        self.connections = set()

    async def connect(self, websocket):
        await websocket.accept()
        self.connections.add(websocket)
Room manager
class RoomManager:
    def __init__(self):
        self.rooms = defaultdict(set)

    async def join(self, room, websocket):
        await websocket.accept()
        self.rooms[room].add(websocket)
Redis Pub/Sub
class RedisPubSub:
    async def publish(self, channel, message):
        await self.redis.publish(
            channel,
            json.dumps(message),
        )

5. Các lỗi thực chiến phải nhấn mạnh khi đi dạy

Thiếu cleanup trong `finally`

Memory tăng dần, online count sai và broadcast dễ gửi vào socket đã chết.

Không bắt `WebSocketDisconnect`

Client đóng tab hoặc rớt mạng làm handler văng lỗi mà không dọn state.

Dùng blocking I/O trong async handler

`requests`, `time.sleep()` hoặc I/O chặn event loop sẽ làm toàn bộ client khác bị lag.

Không có heartbeat

Connection chết ngầm vẫn bị giữ lại do proxy, NAT hoặc mạng di động.

Auth WebSocket sơ sài

Không nên nghĩ middleware REST mặc định đã bảo vệ đủ cho luồng realtime.

Log token ở query string

JWT hoặc token có thể lộ trong access log, APM hoặc lịch sử trình duyệt.

Multi-worker không sync state

`set()` trong memory chỉ đúng khi chạy một process. Scale ngang cần pub/sub bên ngoài.

Không giới hạn message size

Payload quá lớn gây spike RAM, làm server chậm hoặc bị abuse.

Nginx cấu hình sai

Thiếu `Upgrade`, `Connection` hoặc timeout không đủ sẽ làm handshake fail hoặc drop connection.

Thiếu reconnect và resync

Client reconnect nhưng không lấy lại state cũ sẽ làm dashboard hoặc notification bị lệch.

6. Kiến trúc production: FastAPI + Redis + Nginx

1 Browser mở `wss://.../ws/rooms/general` qua Nginx.
2 Nginx forward đúng `Upgrade` và `Connection` tới backend.
3 Gunicorn/Uvicorn chạy nhiều worker ASGI.
4 Redis Pub/Sub đồng bộ message giữa các worker.
5 App áp dụng auth, origin check, validation, heartbeat và cleanup.

`main.py` cần có gì

Auth, rate limit, Pydantic validation, heartbeat, Redis listener, cleanup trong `finally`.

`nginx.conf` cần có gì

`proxy_http_version 1.1`, `Upgrade`, `Connection`, timeout đủ dài và `proxy_buffering off`.

Docker/Gunicorn

Dùng `uvicorn.workers.UvicornWorker` để chạy ASGI app nhiều process một cách chuẩn hơn cho production self-hosted.

Nginx essentials

proxy_set_header Upgrade $http_upgrade;
proxy_set_header Connection $connection_upgrade;
proxy_read_timeout 3600s;
proxy_buffering off;

7. Checklist production

Application

Có auth, origin check, validation, rate limit, heartbeat, reconnect strategy và cleanup.

Infrastructure

Có proxy config đúng, timeout hợp lý, TLS, health check, metrics và graceful shutdown.

State sync

Không dùng in-memory broadcast khi đã chạy nhiều worker hoặc nhiều máy.

Observability

Theo dõi active connections, messages/sec, disconnect reason, queue size và send latency.

8. Khi nào nên và chưa nên dùng WebSocket?

Nên dùng

Chat, live dashboard, trading, multiplayer, collaborative editing, presence và các use case cần hai chiều, latency thấp.

Chưa nên dùng

CRUD thông thường, stream một chiều đơn giản, hệ có tần suất update thấp hoặc cần mô hình scale cực đơn giản.

Kết luận thực dụng: WebSocket với FastAPI không khó ở phần demo, mà khó ở phần vận hành bền vững. Khi dạy chủ đề này, hãy nhấn mạnh rằng production WebSocket bắt đầu từ lifecycle, auth, proxy, sync state và cơ chế phục hồi sau reconnect.
