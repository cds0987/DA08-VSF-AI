1. Nền tảng: Vector Database là gì?

Vector Database là hệ cơ sở dữ liệu tối ưu cho việc lưu trữ, lập chỉ mục và truy vấn embedding vector. Mỗi vector là một biểu diễn số của ý nghĩa ngữ nghĩa, thường dùng cho văn bản, hình ảnh, âm thanh hoặc dữ liệu đa phương thức.

Ví dụ trực quan

“Cách reset mật khẩu?” và “Quên password thì làm sao?” có thể rất khác về từ ngữ, nhưng embedding của chúng thường nằm gần nhau vì cùng biểu diễn một ý định.

Ứng dụng nổi bật

Semantic search, RAG chatbot, image similarity, recommendation, duplicate detection và các hệ truy vấn theo ngữ nghĩa.

Nhu cầu SQL Elasticsearch/BM25 Vector DB
Tìm exact match Tốt Tốt Không phải mục tiêu chính
Tìm theo keyword Kém Rất tốt Trung bình nếu không hybrid
Tìm theo ngữ nghĩa Kém Giới hạn Rất tốt

Điểm cốt lõi: Vector Database không thay thế hoàn toàn keyword search. Trong nhiều hệ production, giải pháp mạnh nhất thường là hybrid search, tức kết hợp semantic và lexical retrieval.
Vector Database Pinecone, Qdrant, Weaviate, Milvus, Chroma. Có API, storage, metadata, replication và filtered search.
Index Library FAISS, Annoy, ScaNN, USearch. Mạnh cho nghiên cứu hoặc engine tự xây, nhưng chưa đủ chức năng của một DB hoàn chỉnh.
DB + Extension pgvector, Elasticsearch kNN, MongoDB Atlas Vector Search. Phù hợp khi muốn tận dụng hạ tầng hiện có.

2. So sánh công nghệ và cách chọn

Không có hệ nào tốt nhất cho mọi bối cảnh. Quyết định nên dựa trên quy mô dữ liệu, loại filter, khả năng vận hành, ngân sách và mức độ gắn kết với hạ tầng sẵn có.

Option Điểm mạnh Điểm cần lưu ý Phù hợp nhất
Pinecone Managed tốt, ít vận hành, scale nhanh Lock-in, chi phí usage Enterprise RAG
Qdrant Payload filter mạnh, dễ làm multi-tenant Cần hiểu rõ cấu hình khi scale SaaS RAG, metadata-heavy
Chroma Dễ khởi đầu, hợp local/prototype Production lớn cần cân nhắc MVP, thử nghiệm nhanh
pgvector Tận dụng Postgres, ACID, JOIN mạnh ANN và scale có giới hạn App đã dùng PostgreSQL
Milvus / Zilliz Scale lớn, nhiều chiến lược index Self-host phức tạp Hệ dữ liệu rất lớn

Managed cloud

Vào production nhanh, có SLA và support, đổi lại dễ tăng chi phí và phụ thuộc vendor.

Self-hosted

Kiểm soát tốt hơn, có thể tối ưu chi phí dài hạn, nhưng cần năng lực vận hành rõ ràng.

Vector extension

Giảm độ phức tạp kiến trúc nếu đội đã dùng PostgreSQL hoặc search engine từ trước.

3. Workflow RAG đầy đủ

Muốn dạy người khác hiểu nhanh, hãy xem Vector Database như một mắt xích trong pipeline retrieval, không phải toàn bộ hệ thống.

1 Documents được đưa vào hệ thống.
2 Tài liệu được chia chunk theo ngữ cảnh phù hợp.
3 Chunk được biến thành embedding vector.
4 Vector và metadata được upsert vào Vector DB.
5 Query của người dùng cũng được embedding.
6 Engine chạy similarity search kèm metadata filter.
7 Kết quả có thể qua bước reranking trước khi đưa vào prompt.
8 LLM dùng context được truy hồi để sinh câu trả lời.
Nếu retrieval yếu, LLM mạnh đến đâu cũng trả lời kém. Trong RAG, lỗi thường bắt đầu từ chunking, embedding hoặc filtering.

4. Ví dụ thực hành nhanh

Ba ví dụ dưới đây đủ để minh họa ba hướng phổ biến: embedded prototype, vector DB production-friendly và Postgres extension.

Chroma
import chromadb
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("sentence-transformers/all-MiniLM-L6-v2")
client = chromadb.PersistentClient(path="./chroma_db")
collection = client.get_or_create_collection(name="docs")

collection.upsert(
    ids=["doc1"],
    documents=["Vector DB lưu embedding để semantic search."],
    embeddings=model.encode(
        ["Vector DB lưu embedding để semantic search."],
        normalize_embeddings=True,
    ).tolist(),
    metadatas=[{"tenant_id": "acme"}],
)
Qdrant
from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams

client = QdrantClient(path="./qdrant_local")
client.recreate_collection(
    collection_name="docs",
    vectors_config=VectorParams(
        size=384,
        distance=Distance.COSINE,
    ),
)
pgvector
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE documents (
  id BIGSERIAL PRIMARY KEY,
  tenant_id TEXT NOT NULL,
  content TEXT NOT NULL,
  embedding vector(384)
);

CREATE INDEX documents_embedding_hnsw_idx
ON documents
USING hnsw (embedding vector_cosine_ops);

5. Các lỗi kinh điển cần nhấn mạnh khi đi dạy

Model mismatch

Dùng model A để index và model B để query. Hệ thống không hỏng ngay nhưng kết quả retrieval sai âm thầm.

Dimension mismatch

Collection cấu hình 1536 chiều nhưng embedding trả về 384 chiều. Đây là lỗi cấu hình rất thường gặp ở giai đoạn đầu.

Quên tenant filter

Đặc biệt nguy hiểm trong SaaS multi-tenant vì có thể gây lộ dữ liệu giữa các khách hàng.

Post-filter sai chỗ

Search top-k toàn bộ trước rồi mới lọc. Kết quả là có thể mất hết các tài liệu đúng của tenant mục tiêu.

Không đo recall

Latency đẹp không có nghĩa là retrieval tốt. Cần đo recall@k, MRR, nDCG và tỉ lệ empty result.

Chunking thiếu chủ đích

Chunk quá lớn làm loãng thông tin; chunk quá nhỏ làm mất ngữ cảnh. Đây là lỗi làm giảm chất lượng RAG rất nhanh.

6. Lộ trình học từ zero đến production

Bước 1. Hiểu embedding qua trực quan hóa

Encode vài trăm câu tiếng Việt và dùng PCA, t-SNE hoặc UMAP để nhìn ra các cụm nghĩa.

Bước 2. Thử FAISS trên local

So sánh Flat, IVF, HNSW và học trade-off giữa recall, latency và memory.

Bước 3. Xây RAG nhỏ bằng Chroma

Học chunking, metadata, persistent storage và truy hồi cơ bản.

Bước 4. Chuyển sang Qdrant hoặc Milvus

Luyện filtered search, payload index, batch upsert và cấu hình collection theo hướng production.

Bước 5. Học pgvector nếu hệ đã có Postgres

Kết hợp vector search với dữ liệu quan hệ và tối ưu truy vấn bằng EXPLAIN ANALYZE.

Bước 6. Thêm hybrid search và reranking

Kết hợp BM25, dense vector, sparse vector và score fusion để tăng chất lượng retrieval.

Bước 7. Thiết kế multi-tenant RAG

Rèn luyện tenant isolation, filter strategy và security tests.

Bước 8. Benchmark trên dữ liệu thật

So sánh các hệ trên cùng workload: recall@5, p95 latency, RAM, disk, ingest throughput.

Bước 9. Production readiness

Hoàn thiện backup, restore, reindex pipeline, model versioning, cost control và incident playbook.

7. Mười nguyên tắc vàng cho production

  • Version hóa embedding Lưu rõ model, dimension, metric và thời điểm tạo.
  • Tenant isolation là bắt buộc Không bao giờ bỏ qua namespace hoặc tenant filter.
  • Benchmark trên dữ liệu thật Dữ liệu synthetic dễ làm sai cảm nhận về hiệu năng.
  • Đo recall song song với latency Retrieval fail thường âm thầm hơn lỗi hệ thống.
  • Thiết kế reindex sớm Đổi model hoặc chunking là chuyện gần như chắc chắn sẽ xảy ra.
  • Dùng batch ingest Insert từng vector vừa chậm vừa tốn chi phí.
  • Index cho field filter Nếu filter nhiều theo tenant, lang, category thì phải lập index.
  • Không chọn DB theo hype Chọn theo workload, không chọn theo benchmark công khai.
  • Có abstraction layer Giảm phụ thuộc sâu vào SDK của một vendor.
  • Tính tổng chi phí sở hữu Bao gồm storage, RAM, read/write, replica, backup và công vận hành.
